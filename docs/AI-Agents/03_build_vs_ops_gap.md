# Build vs Ops 갭 분석: AgentOps의 부상 (2026)

AI 에이전트 생태계가 "만드는 도구"에 편중되어 있다는 주장은 얼마나 사실인가?  
이 문서는 그 갭의 실체와 원인, 그리고 새로 등장하는 "AgentOps" 개념을 분석한다.

---

## 1. Build 편중이 사실인가?

**결론: 사실이다. 그러나 2025-2026년을 기점으로 Ops 도구가 빠르게 성숙 중이다.**

### 빌드 도구의 압도적 우세

2023-2024년에 출시된 도구의 80% 이상이 "빌드" 레이어에 집중되었다.

| 레이어 | 도구 수 (2024 기준) | 성숙도 |
|--------|-----------------|--------|
| 에이전트 프레임워크 | 20+ | 높음 |
| Low-code 빌더 | 15+ | 중-높 |
| 클라우드 런타임 | 5-6 | 중간 |
| 관찰성 / 모니터링 | 6-8 | 낮-중 |
| 평가 프레임워크 | 4-5 | 낮음 |
| 버저닝 / 거버넌스 | 2-3 | 매우 낮음 |

### 왜 이런 편중이 생겼나?

1. **에이전트가 먼저 존재해야 관찰할 것이 생긴다**: 자연스러운 순서이지만, 빌드 도구가 Ops보다 2-3년 앞서 나왔다.
2. **VC 투자 패턴**: "에이전트 빌더"가 더 이해하기 쉬운 스토리. Ops 도구는 "공학 인프라"로 인식돼 투자 유치가 어려웠다.
3. **오픈소스 기여 편중**: 새 에이전트 프레임워크 GitHub 스타를 받기 쉽다. 관찰성 툴링은 상대적으로 덜 화려하다.
4. **기업들의 파일럿 우선**: 대부분 기업이 2023-2024년에 "에이전트 파일럿"에 집중하고, 프로덕션 운영 문제는 나중에 직면한다.

## 2. 전통 MLOps / DevOps와의 갭

### DevOps에서 AI Agent까지: 가정이 모두 깨진다

| DevOps/MLOps의 가정 | AI 에이전트에서 깨지는 이유 |
|---------------------|---------------------------|
| 결정론적 입출력 | 같은 입력 → 다른 출력 (확률적) |
| 단위 테스트 통과/실패 | "올바른 답"이 달라질 수 있어 정의 불가 |
| 코드 배포 = 동작 변경 | 모델 업데이트, 프롬프트 변경, 컨텍스트 변화도 동작 변경 |
| 회귀 테스트로 안전 검증 | 동일 테스트 통과해도 다른 케이스 실패 가능 |
| 롤백 = 이전 코드로 되돌리기 | 에이전트 "상태"는 코드만이 아님 (프롬프트, 메모리, 도구 포함) |
| SLA = 레이턴시 + 가용성 | AI 에이전트는 응답 품질도 SLA에 포함 |

### MLOps와도 다른 점

MLOps는 **모델(파라미터)** 을 관리한다.  
AgentOps는 **자율적 행동(프롬프트 + 워크플로우 + 도구 권한 + 메모리 + 오케스트레이션 로직)** 을 관리한다.

```
MLOps:
  코드 → [모델 학습] → [모델 서빙] → 모니터링
  
AgentOps:
  프롬프트 → [에이전트 런타임] → [도구 실행] → [메모리 업데이트]
       ↑                                              ↓
  피드백 루프 ← 평가 ← 관찰 ← 결과 ← [외부 시스템 상호작용]
```

## 3. AgentOps란 무엇인가

> "DevOps가 소프트웨어를 신뢰할 수 있게 만들었고, MLOps가 모델을 프로덕션에 올렸다면, AgentOps는 에이전트 AI를 신뢰할 수 있게 만든다." — IBM Think 2026

AgentOps는 자율 AI 에이전트의 전체 생명주기를 관리하는 새로운 엔지니어링 규율이다. MLOps와 LLMOps를 확장하되, **자율성**과 **비결정론적 행동**이라는 새로운 차원을 추가한다.

### AgentOps의 6가지 핵심 영역

1. **프롬프트 버저닝 & 거버넌스**
   - 프롬프트를 코드처럼 버전 관리, 변경 추적, 롤백
   - 환경(dev/staging/prod)별 프롬프트 분리

2. **워크플로우 / 오케스트레이션 관리**
   - 에이전트 그래프/플로우를 배포 단위로 관리
   - 플로우 변경의 영향 범위 분석

3. **도구 권한 & 접근 제어**
   - 에이전트가 어떤 도구에 어떤 권한으로 접근하는지 명시적 관리
   - 최소 권한 원칙 적용

4. **런타임 가드레일**
   - 실행 중 안전 규칙 집행 (콘텐츠 필터, 비용 제한, 루프 탈출)
   - 이상 행동 자동 차단 또는 인간 개입 요청

5. **관찰성 & 평가 루프**
   - 프로덕션 트레이스 수집 → 평가 → 개선 사이클
   - 에이전트 "품질 회귀"를 코드 회귀처럼 탐지

6. **롤백 & 생명주기 관리**
   - 에이전트 = 프롬프트 + 플로우 + 메모리 + 도구 설정의 복합 산출물
   - 이 전체를 하나의 버전 단위로 롤백 가능해야 함

## 4. 현재 해결되지 않은 문제들

### 4.1 테스트와 CI/CD

**문제**: 전통 CI/CD는 결정론적 출력을 전제로 한다. `assert output == expected_output` 이 의미없다.

**현재 상태**:
- LLM-as-Judge 패턴이 2026년 표준이 되었지만, 평가 모델 자체의 신뢰성 문제가 남음
- "궤적 테스팅(Trajectory Testing)": 출력 대신 에이전트가 취한 행동 경로를 검증 — 아직 도입률 낮음
- 통계적 테스트(다수 실행 후 평균 성공률): 비용이 높고 CI 파이프라인에 통합하기 어려움

**아직 없는 것**: 에이전트를 위한 표준 CI/CD 파이프라인 패턴. GitHub Actions에 에이전트 품질 게이트를 추가하는 검증된 방법이 없다.

### 4.2 신뢰성 갭

연구에 따르면:
- 단일 실행 성공률: **60%**
- 8회 연속 실행 성공률: **25%** 로 급락

이는 에이전트가 독립적이고 반복 가능해야 하는 프로덕션 환경에서 치명적이다. **신뢰성을 측정하고 보장하는 표준 방법이 아직 없다.**

### 4.3 프롬프트·컨피그 드리프트

모델 업데이트, 프롬프트 변경, 도구 API 변경이 에이전트 동작을 무성고 변경한다.

**문제**: 코드는 변경하지 않았는데 에이전트 동작이 달라지는 "암묵적 변경"
- 모델 제공자의 모델 업데이트
- 벡터 DB의 인덱스 변경 (RAG 결과 변동)
- 외부 API 응답 형식 변화

**아직 없는 것**: 이러한 암묵적 변경을 자동으로 감지하고 알림을 보내는 표준 도구.

### 4.4 비용 예측 불가능성

에이전트는 태스크 복잡도에 따라 토큰 소비가 폭발적으로 증가할 수 있다.
- 동일한 태스크 유형도 입력에 따라 10x~100x 비용 차이
- 루프 탈출 실패 시 무한 토큰 소비

**아직 없는 것**: 에이전트 실행 비용을 사전에 예측하고, 예산 초과 시 자동으로 중단하는 표준 메커니즘.

### 4.5 멀티에이전트 디버깅

단일 에이전트도 디버깅이 어려운데, 여러 에이전트가 협력하면 실패 원인 추적이 극도로 복잡해진다.

**문제**:
- 어느 에이전트의 결정이 최종 실패를 초래했는가?
- 에이전트 간 메시지 전달에서 정보 손실이 있었는가?
- 루프/순환 호출에서 어디서 문제가 발생했는가?

**현재 상태**: 단일 에이전트 트레이싱은 LangSmith/Langfuse 등이 잘 지원하지만, 진정한 멀티에이전트 분산 추적은 아직 미성숙.

### 4.6 거버넌스 & 컴플라이언스

에이전트가 실제 행동(이메일 전송, DB 수정, 외부 API 호출)을 수행하기 시작하면서 새로운 리스크가 생겼다.

**해결되지 않은 문제**:
- 에이전트 행동의 감사 추적(audit trail) 표준 없음
- "에이전트가 한 행동"의 법적 책임 귀속 불명확
- 규제 산업(금융, 의료)에서 에이전트 행동의 컴플라이언스 증명 방법 미정립

## 6. AgentOps 도입 로드맵 (실용 가이드)

### 단계 1: 기초 (지금 당장 시작)
- [ ] 프레임워크에 관찰성 SDK 추가 (LangSmith or Langfuse)
- [ ] 모든 에이전트 실행에 트레이싱 활성화
- [ ] 기본 비용 대시보드 설정

### 단계 2: 평가 루프 구축
- [ ] 에이전트 테스트 데이터셋 구축 (최소 50개 케이스)
- [ ] LLM-as-judge 평가기 설정
- [ ] 평가를 배포 전 필수 체크로 통합

### 단계 3: 버저닝 & 거버넌스
- [ ] 프롬프트를 코드와 함께 Git으로 버전 관리
- [ ] 도구 권한 목록을 명시적으로 문서화 및 리뷰
- [ ] 에이전트 동작 변경에 대한 변경 로그 유지

### 단계 4: 신뢰성 엔지니어링
- [ ] 비용 한도 및 루프 탈출 조건 모든 에이전트에 적용
- [ ] 인간 개입 지점 설계 (HITL)
- [ ] 이상 행동 알림 설정

## 7. 핵심 통찰 요약

1. **빌드 도구는 풍부하다**: 2026년 기준 에이전트를 만드는 도구는 선택지가 너무 많을 정도로 성숙했다.

2. **Ops 도구는 빠르게 따라오고 있다**: 2025-2026년을 기점으로 LangSmith, Langfuse, Arize Phoenix 등이 프로덕션급으로 성숙했다.

3. **그러나 근본적 갭은 여전히 존재한다**:
   - CI/CD 통합 표준이 없다
   - 비결정론적 시스템의 신뢰성을 보장하는 방법이 없다
   - 멀티에이전트 시스템 디버깅이 극도로 어렵다
   - 에이전트 행동의 거버넌스/컴플라이언스 체계가 미성숙하다

4. **"AgentOps"라는 개념이 등장했지만** 아직 표준화된 실천 체계가 없다. 팀마다 다른 방식으로 문제를 해결하고 있다.

5. **2027년 예측**: 가트너는 "2027년 말까지 에이전트 AI 프로젝트의 40% 이상이 취소될 것"이라고 예측한다. 그 이유는 에이전트를 만들 수 없어서가 아니라, **신뢰할 수 있게 운영하지 못해서**다.
